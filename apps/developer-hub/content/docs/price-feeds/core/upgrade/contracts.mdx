Pyth has three kinds of on-chain contracts. This page explains how to tell them apart and lists all three side by side, per chain.

<Callout type="info">
  **If you remember one thing:** Pyth Core and Pyth Pro are two different
  products with two different shapes. The "upgraded" contract is not Pro. It's
  the next generation of the Pyth Core contract you already use: same
  interface, same feed IDs, now with the richer data and customizability that
  Pro introduced.
</Callout>

|                          | What it is                                                                                                                       | Built for                                                                                |
| ------------------------ | --------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------- |
| **Pyth Core (current)**  | The contract you integrated against today. Keeps a **shared price on-chain** that any contract can read                            | Existing integrations                                                                      |
| **Pyth Core (upgraded)** | The **same interface and feed IDs** at a new address, with richer data and more customization. Still keeps the shared price on-chain | Upgrading your existing integration before July 31 (a [drop-in address swap](/price-feeds/core/upgrade/preparing)) |
| **Pyth Pro**             | A different model: **no price stored on-chain**. Each transaction carries its own millisecond-fresh signed price, verified on the spot | Latency-critical trading priced per transaction: perps fills, RFQ and quote-based execution |

## The real difference between upgraded Core and Pro

**Upgraded Core keeps one shared price on-chain.**

- Your contract calls [`getPriceNoOlderThan`](/price-feeds/core/api-reference) and reads the **same canonical price** every other protocol on the chain sees.
- That's the shape **lending markets, CDPs, and vaults** are built around: liquidations triggered by third parties, share prices computed in view functions, protocols composing by reading the same stored value.
- [Sponsored push feeds](/price-feeds/core/push-feeds) keep the price fresh, so passive protocols integrate with a **single read**.
- The upgrade changes none of this: **same interface, same feed IDs**, better data behind them.

**Pyth Pro attaches the price to each transaction.**

- **Nothing is stored on-chain.** Every transaction that needs a price carries a freshly signed one, [verified on the spot](/price-feeds/pro/integrate-as-consumer).
- Each trade executes on a price that is **milliseconds old**: what perps fills and RFQ execution need, and what no stored price can provide.
- The trade-off: **no shared price** for other contracts to read.

<Callout>
  **One question decides it.** Do other contracts, third-party liquidators, or
  view functions need to read your price on-chain? If yes, you want **Core**.
  If not, take a look at [**Pyth Pro**](/price-feeds/pro).
</Callout>

## What happens on July 31, 2026

- Your **current contract is upgraded in place**. It starts accepting the new data payloads automatically. You keep your address and your feed IDs.\*
- `hermes.pyth.network` is **redirected to the upgraded backend**. The URL keeps working, but every request needs a [Pyth API Key](https://pythdata.app/signup?utm_source=developer-hub&utm_campaign=core-upgrade&utm_content=contract-addresses) from that day on.
- From then on, "current" and "upgraded" are the same thing, and that distinction disappears. What remains is Core vs Pro: the same upgraded data, either stored on-chain for everyone to read, or attached fresh to each transaction.

\* Except on Sui, where the in-place upgrade isn't possible; see the [Sui upgrade guide](/price-feeds/core/upgrade/preparing/sui).

## The two mistakes this page exists to prevent

1. **Swapping a Core integration to a Pro address.** Pro has a different interface, different feed IDs, and no stored price to read; nothing will work. If you're upgrading Core, you want the **upgraded Core** address (listed below).
2. **Swapping the contract address without moving your data source, or vice versa.** Each endpoint's payloads verify only on its own contract generation. Swap both together, or wait and let July 31 switch both for you.

**Where to go next:** the [upgrade guide](/price-feeds/core/upgrade/preparing) for the how, or the [Pyth Pro docs](/price-feeds/pro) if per-transaction pricing is what you need. All addresses, all three kinds, are below.
